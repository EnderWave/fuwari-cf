# 第2章：信息的表示和处理

  

## 一、信息的本质：一切都是位（Bits）

  

- 计算机中的所有信息——整数、浮点数、字符、机器指令、地址、控制信号——均由二进制位（0 和 1）编码。

- 不同的**解释方式（编码方案）** 将相同的位模式映射为不同的语义。例如，同一个 32 位模式 `0x41400000`，按整数解释是 `1094713344`，按 IEEE 754 浮点数解释是 `12.0`，按机器指令解释可能是 `add` 指令的一部分。

- **为什么用二进制？** 电子电路天然支持两种稳定状态（高电压 / 低电压、导通 / 截止），用 0 和 1 表示最可靠、抗噪声能力最强。

## 二、进制基础：二进制、十进制、十六进制

  

### 2.1 二进制（Binary，基数为 2）

- 每一位的权值为 $2^i$（从右往左，$i=0,1,2,\dots$）。

- 例：$1011_2 = 1\times2^3 + 0\times2^2 + 1\times2^1 + 1\times2^0 = 8+0+2+1 = 11_{10}$。

  

### 2.2 十六进制（Hexadecimal，基数为 16）

- 用 0~9 表示 0~9，用 A~F（或 a~f）表示 10~15。

- C 语言中以 `0x` 或 `0X` 为前缀，如 `0xFA1D`、`0x3B6D`。

- **十六进制 ↔ 二进制 转换技巧**：每 1 位十六进制数对应 4 位二进制数（因为 $16=2^4$）。从小数点开始向两边，每 4 位一组，不足补 0。

  - 例：`11101101101101₂` → 分组 `0011 1011 0110 1101` → `0x3B6D`。

  - 反方向：`0xA5` → `1010 0101₂`。

  

### 2.3 十进制 ↔ 二进制 转换（整数部分）

- **除 2 取余法**：不断除以 2，倒序取余数。

  - 例：$15213_{10} \to 二进制$：

    - 15213 ÷ 2 = 7606 余 1

    - 7606 ÷ 2 = 3803 余 0

    - … 最终得到 `11101101101101₂`。

  

### 2.4 字长（Word Size）与数据大小

- **字长**是指针数据的标称大小，决定了 CPU 一次能处理的最大整数位数，也决定了虚拟地址空间的范围：

  - 32 位字长 → 地址范围 $0 \sim 2^{32}-1$ = 4 GB。

  - 64 位字长 → 地址范围 $0 \sim 2^{64}-1$ $\approx$ 16 EB（1.84×10¹⁹ 字节）。

- C 语言常用数据类型在 64 位 x86-64 上的典型大小（字节）：

  - `char`: 1, `short`: 2, `int`: 4, `long`: 8, `float`: 4, `double`: 8, `pointer`: 8。

## 三、位级运算（Bitwise Operations）

  

### 3.1 布尔代数四则运算（逐位独立）

| 操作符 | 名称 | 规则 |

| :--- | :--- | :--- |

| `&` | 按位与 | 两者为 1 则结果为 1，否则为 0 |

| `|` | 按位或 | 至少一个为 1 则结果为 1 |

| `~` | 按位取反 | 0 ↔ 1 互换 |

| `^` | 按位异或 | 不同为 1，相同为 0（不进位加法） |

  

### 3.2 位向量的集合应用

- 宽度为 $w$ 的位向量可以表示集合 $\{0,1,\dots,w-1\}$，其中第 $j$ 位为 1 表示元素 $j$ 在集合中。

- 例：`01101001`（从高位到低位）表示集合 $\{0, 3, 5, 6\}$。

- 集合运算对应位运算：

  - `&` → 交集

  - `|` → 并集

  - `^` → 对称差

  - `~` → 补集

  

### 3.3 C 语言示例（char 类型）

- `~0x41` → `0xBE`（因为 0x41 = 0100 0001，取反为 1011 1110）

- `0x69 & 0x55` → `0x41`（0110 1001 & 0101 0101 = 0100 0001）

- `0x69 | 0x55` → `0x7D`

- `0x69 ^ 0x55` → `0x3C`

## 四、逻辑运算（Logical Operations）—— 极易与位运算混淆

  

### 4.1 核心规则

- `&&`（逻辑与）、`||`（逻辑或）、`!`（逻辑非）

- 将 **0 视为假（False）**，将**任何非 0 值视为真（True）**，返回值始终为 **0 或 1**。

- **短路求值（Short-circuit）**：

  - `A && B`：若 A 为假，则 B 不计算（整体必假）。

  - `A || B`：若 A 为真，则 B 不计算（整体必真）。

  - 典型应用：`p && *p` 可安全避免空指针解引用（若 `p == 0`，则 `*p` 不执行）。

  

### 4.2 与位运算的本质区别（最高频陷阱）

| 表达式 | 结果 | 解释 |

| :--- | :--- | :--- |

| `0x69 && 0x55` | `0x01` | 两个非 0 值逻辑与为真 → 1 |

| `0x69 & 0x55` | `0x41` | 逐位与，保留重叠位 |

| `!0x41` | `0x00` | 非 0 值取逻辑非 → 假 |

| `~0x41` | `0xBE` | 逐位取反 |

  

**常见错误**：将 `&` 写成 `&&`，或将 `|` 写成 `||`，导致语义完全改变。

## 五、移位操作（Shift Operations）

  

### 5.1 左移 `<<`

- 将位向量向左移动 $k$ 位，**高位丢弃，低位补 0**。

- 等价于乘以 $2^k$（对有符号数也成立，前提是不溢出）。

  

### 5.2 右移 `>>` —— 必须区分两种

| 类型 | 高位补什么 | 适用场景 | 等价于 |

| :--- | :--- | :--- | :--- |

| **逻辑右移** | 补 0 | 无符号数 | 除以 $2^k$ 并向下取整 |

| **算术右移** | 补**原符号位**（正数补 0，负数补 1） | 有符号数 | 保持符号，但向负无穷取整（注意偏差） |

  

### 5.3 未定义行为

- 移位量 $k < 0$ 或 $k \ge \text{字长}$ 时，结果依赖于具体编译器实现，属未定义行为，应避免。

## 六、整数编码：无符号（Unsigned）与补码（Two's Complement）

  

### 6.1 无符号数编码（B2U）

对于位向量 $\vec{x} = [x_{w-1}, x_{w-2}, \dots, x_0]$：

$$B2U(\vec{x}) = \sum_{i=0}^{w-1} x_i \cdot 2^i$$

- 范围：$0 \sim 2^w - 1$。

- 例如（w=4）：`1010` → $1\times8 + 0\times4 + 1\times2 + 0\times1 = 10$。

  

### 6.2 补码编码（B2T）—— 有符号数的国际标准

$$B2T(\vec{x}) = -x_{w-1} \cdot 2^{w-1} + \sum_{i=0}^{w-2} x_i \cdot 2^i$$

- **最高位（MSB）为符号位**：1 表示负数，0 表示非负数。

- 范围：$-2^{w-1} \sim 2^{w-1} - 1$（**非对称**，负数比正数多一个）。

- 例如（w=4）：`1010` → $-1\times8 + 0\times4 + 1\times2 + 0\times1 = -6$。

### 6.4 四种极值（w 位）

| 名称 | 二进制（w 位） | 十六进制（w=16 示例） | 数值 |

| :--- | :--- | :--- | :--- |

| **UMax** | `111...1`（全 1） | `0xFFFF` | $2^w - 1$ |

| **TMax** | `011...1` | `0x7FFF` | $2^{w-1} - 1$ |

| **TMin** | `100...0` | `0x8000` | $-2^{w-1}$ |

| **-1** | `111...1`（全 1） | `0xFFFF` | -1（补码） |

- 观察：$\lvert TMin \rvert = TMax + 1$（非对称）；$UMax = 2 \times TMax + 1$。

## 七、有符号 ↔ 无符号的转换（位模式不变，解释改变）

  

### 7.1 映射规则（w 位）

- **T2U(x)**（有符号 → 无符号）：

  - 若 $x \ge 0$，则 $T2U(x) = x$；

  - 若 $x < 0$，则 $T2U(x) = x + 2^w$（负数变为巨大的正数）。

- **U2T(u)**（无符号 → 有符号）：

  - 若 $u \le TMax$，则 $U2T(u) = u$；

  - 若 $u > TMax$，则 $U2T(u) = u - 2^w$（大正数变为负数）。

  

### 7.2 混合运算中的隐式转换 —— **全书最危险陷阱**

- **规则**：当有符号数与无符号数同时出现在一个表达式中（包括比较操作 `<`, `>`, `==`, `<=`, `>=`），**有符号数会被隐式转换为无符号数**。

- **经典笔试题（PDF P28，w=32）**：

  - `-1 < 0U`：`-1` → `4294967295U`，比较 `4294967295U < 0U` 为假 → 结果为 **0（False）**。

  - `-1 > -2`：纯有符号比较，`-1 > -2` 为真 → 结果为 **1（True）**。

  - `(unsigned)-1 > -2`：左 `4294967295U`，右 `-2` → `4294967294U`，`4294967295 > 4294967294` 为真 → **True**。

  - `2147483647U > -2147483648`：右 `-2147483648` → `2147483648U`，比较 `2147483647U > 2147483648U` 为假 → **False**。

  - `2147483647 > (int)2147483648U`：右 `2147483648U` 强制转 int → `-2147483648`，比较 `2147483647 > -2147483648` 为真 → **True**。

## 八、扩展（Expanding）与截断（Truncating）

  

### 8.1 扩展（从 w位扩展到 w+k 位）

- **无符号扩展（零扩展）**：高位全部补 0。

- **有符号扩展（符号扩展）**：高位全部复制**原符号位**（补 k 个副本）。

  - 例：`short x = 15213`（`0x3B6D`）→ `int ix` → `0x00003B6D`（值不变）。

  - 例：`short y = -15213`（`0xC493`）→ `int iy` → `0xFFFFC493`（值仍为 -15213，因高位补 1 保持了补码的数值）。

- 结论：扩展后数值保持不变（有符号用符号扩展，无符号用零扩展）。

  

### 8.2 截断（从 w+k 位缩减到 w 位）

- 直接丢弃高 k 位，保留低 w 位，重新解释。

- **无符号截断**：等价于原值 $\bmod 2^w$。

- **有符号截断**：先按模运算取低位，再用补码解释（可能改变符号）。

  - 例：`int x = 100000`（超出 `short` 范围），截断为 `short` 后，值变为 $100000 \bmod 65536 = 34464$，作为有符号数 $34464 - 65536 = -31072$。

## 九、整数运算：加法、乘法与溢出

  

### 9.1 无符号加法

- 执行 w 位二进制加法，若最高位产生进位则**丢弃**（即模 $2^w$）。

- 溢出检测：若 `s < u` 或 `s < v`，则发生了溢出（进位丢失）。

  

### 9.2 补码加法

- 位级行为与无符号加法**完全相同**（都是同样的二进制加法），但解释为有符号值时：

  - **正溢出**（结果 $\ge 2^{w-1}$）：结果变为负数（减去 $2^w$）。

  - **负溢出**（结果 $< -2^{w-1}$）：结果变为正数（加上 $2^w$）。

- 溢出检测：

  - 正溢出：`u > 0 && v > 0 && s <= 0`。

  - 负溢出：`u < 0 && v < 0 && s >= 0`。

  

### 9.3 乘法

- 两个 w 位整数相乘，精确结果可能需要 $2w$ 位。C 语言直接截断低 w 位（相当于模 $2^w$）。

- **关键结论**：无符号乘法和补码乘法的**低 w 位完全相同**（因位运算一致）。

  

### 9.4 用移位优化乘除法（编译器自动生成）

- **乘以 $2^k$**：`x << k`（有符号/无符号均适用）。

- **无符号除以 $2^k$**：**逻辑右移** `>> k`，向零舍入（完美契合）。

- **有符号除以 $2^k$（难点）**：

  - 直接使用**算术右移** `>> k` 是**向负无穷舍入**（例如 `-7 >> 1` 得 `-4`，而实际整除应得 `-3`）。

  - **正确向零舍入的偏置技巧**（C 语言实现）：

    $$(x + (1 << k) - 1) >> k \quad (\text{当 } x < 0 \text{ 时})$$

    - 例：`-15213 / 8`。直接右移得 `-1902`（偏差），正确做法：`(-15213 + 7) >> 3 = -1901` 。

## 十、内存中的表示：字节序（Endianness）—— 笔试必考

  

### 10.1 小端法（Little Endian，x86/ARM）

- 将**最低有效字节（LSB）** 存储在**最低地址**。

  

### 10.2 大端法（Big Endian，网络字节序/老旧服务器）

- 将**最高有效字节（MSB）** 存储在最低地址。

  

### 10.3 完整图解

变量 `int x = 0x01234567`，起始地址 `0x100`：

  

| 地址 | 小端存储（字节内容） | 大端存储（字节内容） |

| :--- | :--- | :--- |

| `0x100` | `0x67`（LSB） | `0x01`（MSB） |

| `0x101` | `0x45` | `0x23` |

| `0x102` | `0x23` | `0x45` |

| `0x103` | `0x01`（MSB） | `0x67`（LSB） |

  

### 10.4 `show_bytes` 函数

- 将任意指针强制转换为 `unsigned char*`，逐字节打印内存内容。

- 在 x86 小端机器上，`int A = 15213`（`0x3B6D`）的字节输出为：`6D 3B 00 00`（低地址放低字节）。

  

### 10.5 字符串不受字节序影响

- 因为每个字符独立占 1 字节，按顺序存储，末尾 `\0`（0x00）终止。

## 十一、浮点数（IEEE 754）—— 全书最深理论

  

### 11.1 二进制小数基础

- 小数点后第 $i$ 位的权值为 $2^{-i}$。

- 例：`1011.101₂ = 8 + 2 + 1 + 0.5 + 0.125 = 11.625`。

- **局限性**：只能精确表示形如 $x/2^k$ 的数（如 $1/2$, $3/4$）；$1/3$、$1/5$、$1/10$ 在二进制中为无限循环小数（如 $0.1_{10} = 0.0001100110011\dots_2$）。

  

### 11.2 IEEE 754 标准格式

$$(-1)^s \times M \times 2^E$$

- **符号位 s**：0 正，1 负。

- **尾数 M**：规格化时为 `1.frac`；非规格化时为 `0.frac`。

- **指数 E**：通过**偏置（Bias）** 编码，$E = Exp - Bias$。

  

### 11.3 精度选项

| 精度 | 总位数 | s | exp | frac | Bias |

| :--- | :--- | :--- | :--- | :--- | :--- |

| 单精度（float） | 32 | 1 | 8 | 23 | 127 |

| 双精度（double） | 64 | 1 | 11 | 52 | 1023 |

  

### 11.4 三类编码状态（核心）

| 类型 | `exp` 编码 | 实际指数 E | 尾数 M | 用途 |

| :--- | :--- | :--- | :--- | :--- |

| **规格化** | 非全 0 且非全 1 | `Exp - Bias` | `1.frac`（隐含前导 1） | 绝大多数普通数 |

| **非规格化** | 全 0 | `1 - Bias`（固定） | `0.frac` | 表示 ±0 及接近 0 的极小值（平滑过渡） |

| **特殊值** | 全 1 | - | frac=0 → ±∞；frac≠0 → NaN | 溢出、非法操作（0/0, ∞-∞） |

  

### 11.5 浮点数手算编码完整步骤（必考）

将 `15213.0` 转为单精度 float：

1. **转二进制整数**：`15213` → `11101101101101₂`（13 位）。

2. **标准化**（写成科学计数法）：$1.1101101101101_2 \times 2^{13}$（小数点左移 13 位）。

3. **符号位**：$s = 0$（正数）。

4. **指数 Exp**：

   - 实际指数 $E = 13$

   - 单精度 Bias = 127

   - $Exp = 13 + 127 = 140 = 10001100_2$（8 位）

5. **尾数 frac**：

   - 小数部分 `1101101101101`，后面补 `0` 至 23 位 → `11011011011010000000000`。

6. **拼接机器码（32 位）**：

   - `0 | 10001100 | 11011011011010000000000`

   - 分组为十六进制：`0100 0110 0110 1101 1011 0100 0000 0000` → **`0x466DB400`**。

  

### 11.6 舍入模式（默认：向偶数舍入）

- 当结果恰好位于两个可表示数的中间时，舍入到最低有效位为 **0** 的那个数（偶数）。

- **二进制舍入例题**：舍入到最近 $1/4$（保留 2 位小数）。

  - `10.11100₂`（2.875）：中间值，向上舍入至 `11.00₂`（3）。

  - `10.10100₂`（2.625）：中间值，向下舍入至 `10.10₂`（2.5），因为 LSB 为 0。

  

### 11.7 浮点运算的反直觉性质（必考）

- **可交换性**：加法和乘法满足交换律（$a+b = b+a$）。

- **结合律**：**不成立**（因舍入）。反例：$(3.14 + 1e10) - 1e10 = 0$，而 $3.14 + (1e10 - 1e10) = 3.14$。

- **分配律**：**不成立**。反例：$1e20 \times (1e20 - 1e20) = 0.0$，而 $1e20 \times 1e20 - 1e20 \times 1e20 = NaN$。

  

### 11.8 类型转换规则

- `double/float → int`：**截断小数部分**（向零舍入）。溢出或 NaN 时结果未定义（常见为 `TMin`）。

- `int → double`：**精确转换**（double 有 53 位有效位数，足以精确表示所有 32 位 int）。

- `int → float`：可能**舍入**（float 仅 24 位有效位数，超出部分四舍五入）。

## 十二、现实灾难案例（深刻教训）

  

### 12.1 Ariane 5 火箭爆炸（1996）

- **原因**：将 64 位浮点数（水平速率）强制转换为 16 位有符号整数，数值超出范围触发**溢出异常**，导致导航计算机崩溃、火箭自毁。

- **教训**：跨类型缩小转换（尤其是浮点转整数）必须严格验证取值范围。

  

### 12.2 爱国者导弹定位失败（1991）

- **原因**：将 0.1 用 24 位定点小数近似（`0.00011001100110011001100₂`），误差为 $9.54 \times 10^{-8}$。

- **后果**：系统运行 100 小时后，时间偏差 $9.54\times10^{-8} \times 3{,}600{,}000 \approx 0.343$ 秒。飞毛腿导弹速度 2000 m/s，导致**定位偏差约 687 米**，军营被炸毁。

- **改进对比**：若使用 `float`（32位）表示 0.1，误差仅 $5.96\times10^{-9}$，100 小时后偏差约 **43 米**，精度提升 16 倍。若用 32 位定点小数，偏差仅 **0.67 米**。

- **启示**：浮点数精度不足会累积产生灾难性误差；整数乘以常数时，优先使用**定点整数乘法 + 移位**（更快更准）。

## 十三、C 语言无符号循环陷阱（PDF P52~53）

  

**错误写法**：

```c

unsigned i;

for (i = cnt - 2; i >= 0; i--)

    a[i] += a[i+1];

```

- 问题：`i >= 0` 永远为真（无符号数永远非负），导致死循环。

  

**正确写法**：

```c

unsigned i;

for (i = cnt - 2; i < cnt; i--)

    a[i] += a[i+1];

```

- 原理：当 `i = 0` 时，`i--` 变为 `UMax`，条件 `UMax < cnt` 为假，循环退出。

  

**更佳写法**（使用 `size_t`）：

```c

size_t i;

for (i = cnt - 2; i < cnt; i--)

    a[i] += a[i+1];

```

- `size_t` 是无符号类型，长度与字长一致，代码更健壮。

# 第3章：程序的机器级表示

  

## 一、RISC-V 概述与设计哲学

  

### 1.1 什么是指令集体系结构（ISA）

- **ISA（Instruction Set Architecture）** 是计算机硬件与软件之间的接口规范，定义了处理器能够执行的全部指令、寄存器、寻址方式、数据类型等。

- RISC-V（发音为"RISC-five"）是一个**开放、免费**的RISC（精简指令集计算机）ISA，由加州大学伯克利分校发起，现由RISC-V基金会维护。

- RISC-V 的设计目标是成为一个通用ISA，适用于从嵌入式微控制器到高性能服务器的一切计算场景。

  

### 1.2 RISC-V 三大设计原则

| 原则 | 含义 | 体现 |

| :--- | :--- | :--- |

| **1. 规律性简化设计** | 指令格式规整，操作数数量固定 | 所有算术指令均为三操作数形式：`add a, b, c`（a = b + c） |

| **2. 越小越快** | 硬件资源越少，访问速度越快 | 仅提供32个通用寄存器，寄存器的访问速度远高于内存 |

| **3. 好的设计需要好的折中** | 不同指令格式虽有差异，但保持统一长度 | 所有指令均为32位定长，简化取指；立即数字段在不同格式中位置错开，以保证rs1/rs2字段位置固定 |

  

### 1.3 RISC-V 通用寄存器与调用约定（ABI）—— **必须熟记**

- RISC-V 提供了 **32 个通用寄存器** `x0` ~ `x31`，每个寄存器宽度为 XLEN（RV32 为32位，RV64为64位）。

- `x0` 被硬连线为常数 **0**，写入 `x0` 不会产生任何效果（结果被丢弃）。

- 下表列出了寄存器的**别名**及其在过程调用中的约定用途：

  

| 寄存器 | ABI 名称 | 用途 | 谁负责保存（若使用） |

| :--- | :--- | :--- | :--- |

| `x0` | `zero` | 硬连线常数 0 | N/A（不可写） |

| `x1` | `ra` | 返回地址（`jal` 指令自动存入） | 调用者（若需跨调用保留） |

| `x2` | `sp` | 栈指针（指向栈顶，向低地址增长） | 被调用者 |

| `x3` | `gp` | 全局指针（用于访问静态数据段） | — |

| `x4` | `tp` | 线程指针（用于线程局部存储） | — |

| `x5`~`x7` | `t0`~`t2` | **临时寄存器**（被调用者无需保存） | 调用者（若调用后还需使用） |

| `x8` | `s0` / `fp` | 保存寄存器 / 帧指针 | 被调用者 |

| `x9` | `s1` | 保存寄存器 | 被调用者 |

| `x10`~`x11` | `a0`~`a1` | 函数参数 / 返回值 | 调用者（若需保留） |

| `x12`~`x17` | `a2`~`a7` | 函数参数 | 调用者（若需保留） |

| `x18`~`x27` | `s2`~`s11` | 保存寄存器 | 被调用者 |

| `x28`~`x31` | `t3`~`t6` | 临时寄存器 | 调用者 |

  

**核心区别**：

- **临时寄存器（`t*`）**：被调用函数可以随意使用，无需保存和恢复。如果调用者需要在函数调用后继续使用这些寄存器的值，调用者必须在调用前自行保存（压栈）。

- **保存寄存器（`s*`）**：被调用函数如果使用了这些寄存器，必须在函数开头保存（压栈），在函数返回前恢复（出栈），以保证调用者看到的值不变。

  
  

## 二、寻址方式与数据传送

  

### 2.1 寄存器寻址

- 操作数存放在寄存器中，指令直接指定寄存器编号。

- 例：`add x5, x6, x7` → `x5 = x6 + x7`。

  

### 2.2 立即数寻址

- 操作数在指令中直接给出（立即数）。

- 例：`addi x5, x6, 100` → `x5 = x6 + 100`。

- **立即数位宽**：I型指令的立即数为 **12 位有符号数**，范围：`-2048` ~ `2047`。在参与运算前会进行**符号扩展**到 XLEN 位。

  

### 2.3 基址寻址（内存访问的唯一方式）

- RISC-V 采用 **加载/存储架构（Load/Store Architecture）**：只有加载（Load）和存储（Store）指令能访问内存；算术指令只能操作寄存器。

- **加载指令**：将内存中的数据读到寄存器。

  - `lw rd, offset(rs1)`：从地址 `rs1 + offset` 处读取 **4 字节（字）** 到 `rd`。

  - `lh / lhu`：读取 **2 字节（半字）**，带符号扩展 / 零扩展。

  - `lb / lbu`：读取 **1 字节**，带符号扩展 / 零扩展。

  - `ld`（RV64）：读取 **8 字节（双字）**。

- **存储指令**：将寄存器中的数据写入内存。

  - `sw rs2, offset(rs1)`：将 `rs2` 的**低 4 字节**写入地址 `rs1 + offset` 处。

  - `sh / sb`：写入 2 字节 / 1 字节（取寄存器低端）。

  

**示例（PDF P10）**：

C代码：`A[12] = h + A[8];`（假设 `h` 在 `x21`，数组 `A` 的基址在 `x22`，`int` 占 4 字节）

```assembly

lw x9, 32(x22)   # x9 = A[8]（偏移量为 8 * 4 = 32）

add x9, x21, x9  # x9 = h + A[8]

sw x9, 48(x22)   # A[12] = x9（偏移量为 12 * 4 = 48）

```

  

**易错点**：内存地址以**字节**为单位，而 `int` 占 4 字节。因此数组下标为 `i` 的元素，其字节偏移量是 `i * 4`（而不是 `i`）。

  

### 2.4 寄存器 vs 内存的速度差距

| 存储位置 | 访问延迟（约） | 相对速度 |

| :--- | :--- | :--- |

| 寄存器（x0~x31） | 0 ~ 1 个时钟周期 | **最快** |

| L1 Cache | 2 ~ 4 个时钟周期 | 约 5~10 倍慢于寄存器 |

| 主存（DRAM） | 100 ~ 300 个时钟周期 | 约 200 倍慢于寄存器 |

- **编译器策略**：尽可能将频繁使用的变量分配在寄存器中（寄存器分配优化）。当寄存器不够用时，将暂时不用的变量“溢出（spill）”到内存栈中。

  
  

## 三、指令格式与机器编码（笔试常考）

  

RISC-V 所有指令均为 **32 位定长**，分为六种格式（R、I、S、SB、U、UJ）。关键设计点：**`rs1` 和 `rs2` 字段在所有格式中位于相同位置**（`rs1` 在 bits 19-15，`rs2` 在 bits 24-20），这简化了指令译码器的硬件设计。

  

### 3.1 R 型格式（寄存器-寄存器运算）

- **字段**：`funct7(7) | rs2(5) | rs1(5) | funct3(3) | rd(5) | opcode(7)`

- **用途**：`add`、`sub`、`and`、`or`、`xor`、`sll`、`srl`、`sra` 等。

- **编码示例（PDF P21）**：`add x9, x20, x21`

  - `opcode`（0110011）：表示 R 型算术运算

  - `rd`（01001）：目标寄存器 x9

  - `funct3`（000）：加法

  - `rs1`（10101）：源寄存器 x20

  - `rs2`（10100）：源寄存器 x21

  - `funct7`（0000000）：加法的扩展码

  - 二进制拼接：`0000001 0101 1010 000 01001 0110011` → 十六进制 **`0x015A04B3`**。

  

### 3.2 I 型格式（立即数运算 / 加载）

- **字段**：`immediate[11:0](12) | rs1(5) | funct3(3) | rd(5) | opcode(7)`

- **用途**：`addi`、`andi`、`ori`、`xori`、`slli`、`srli`、`srai`、`lw`、`lh`、`lb`、`jalr` 等。

- 立即数在指令中**符号扩展**至 XLEN 位。

  

### 3.3 S 型格式（存储）

- **字段**：`imm[11:5](7) | rs2(5) | rs1(5) | funct3(3) | imm[4:0](5) | opcode(7)`

- **用途**：`sw`、`sh`、`sb`。

- **难点**：立即数被**拆分**为高位部分（bits 31-25）和低位部分（bits 11-7）。这样设计是为了让 `rs1` 和 `rs2` 的位置与 R 型保持一致（bits 24-20 和 19-15 仍是寄存器字段）。

  

### 3.4 SB 型格式（条件分支）

- **字段**：`imm[12,10:5](7) | rs2(5) | rs1(5) | funct3(3) | imm[4:1,11](5) | opcode(7)`

- **用途**：`beq`、`bne`、`blt`、`bge`、`bltu`、`bgeu`。

- 立即数表示**相对于 PC 的偏移量（以 2 字节为单位）**，即实际跳转地址 = `PC + imm × 2`。偏移范围为 `±4KB`（12 位有符号 × 2）。

  

### 3.5 UJ 型格式（无条件跳转）

- **字段**：`imm[20,10:1,11,19:12](20) | rd(5) | opcode(7)`

- **用途**：`jal`（跳转并链接）。

- 立即数同样以 2 字节为单位，跳转范围达 `±1MB`（20 位有符号 × 2）。

  

### 3.6 U 型格式（加载高位立即数）

- **字段**：`immediate[31:12](20) | rd(5) | opcode(7)`

- **用途**：`lui`（Load Upper Immediate），将 20 位立即数加载到目标寄存器的 bits 31-12，低 12 位清零。

- 配合 `addi` 可构造任意 32 位常数。

  
  

## 四、算术与逻辑运算指令集

  

### 4.1 算术运算

| 指令 | 格式 | 功能 |

| :--- | :--- | :--- |

| `add rd, rs1, rs2` | R | `rd = rs1 + rs2` |

| `addi rd, rs1, imm` | I | `rd = rs1 + imm`（符号扩展） |

| `sub rd, rs1, rs2` | R | `rd = rs1 - rs2`（无立即数版本） |

| `lui rd, imm` | U | `rd = imm << 12`（低 12 位清零） |

  

### 4.2 逻辑运算

| 指令 | 格式 | 功能 | 典型用途 |

| :--- | :--- | :--- | :--- |

| `and rd, rs1, rs2` / `andi rd, rs1, imm` | R/I | 按位与 | **掩码**：清零某些位 |

| `or rd, rs1, rs2` / `ori rd, rs1, imm` | R/I | 按位或 | **置位**：将某些位置为 1 |

| `xor rd, rs1, rs2` / `xori rd, rs1, imm` | R/I | 按位异或 | **翻转**某些位（与全 1 异或等价于取反） |

| `slli rd, rs1, shamt` | I | 逻辑左移 | 乘以 $2^{shamt}$（低位补0） |

| `srli rd, rs1, shamt` | I | 逻辑右移 | 无符号除以 $2^{shamt}$（高位补0） |

| `srai rd, rs1, shamt` | I | 算术右移 | 有符号除以 $2^{shamt}$（高位补符号位） |

  

**易错点**：`slli` / `srli` / `srai` 的移位量 `shamt` 只有**低 5 位**有效（RV32）或**低 6 位**有效（RV64）。若移位量 ≥ 字长，结果未定义。

  
  

## 五、控制流：分支与循环的汇编翻译

  

### 5.1 条件分支指令

| 指令 | 条件 | 说明 |

| :--- | :--- | :--- |

| `beq rs1, rs2, label` | `rs1 == rs2` | 相等则跳转 |

| `bne rs1, rs2, label` | `rs1 != rs2` | 不等则跳转 |

| `blt rs1, rs2, label` | `rs1 < rs2` | **有符号**小于则跳转 |

| `bge rs1, rs2, label` | `rs1 >= rs2` | **有符号**大于等于则跳转 |

| `bltu rs1, rs2, label` | `rs1 < rs2` | **无符号**小于则跳转 |

| `bgeu rs1, rs2, label` | `rs1 >= rs2` | **无符号**大于等于则跳转 |

  

**难点**：`blt` / `bge` 与 `bltu` / `bgeu` 的比较逻辑不同。对于有符号数 `-1`（0xFFFFFFFF）和 `1`（0x00000001），`blt` 认为 `-1 < 1` 成立，而 `bltu` 认为 `0xFFFFFFFF > 0x00000001` 不成立。

  

### 5.2 If-Else 语句翻译

**示例（PDF P31）**：

```c

if (i == j) f = g + h;

else f = g - h;

```

假设变量映射：`i→x22, j→x23, f→x19, g→x20, h→x21`

汇编逻辑：

```assembly

    bne x22, x23, Else   # i != j → 跳到 Else

    add x19, x20, x21    # f = g + h

    beq x0, x0, Exit     # 无条件跳转到 Exit（beq x0,x0 永远成立）

Else:

    sub x19, x20, x21    # f = g - h

Exit:

```

  

### 5.3 While 循环翻译

**示例（PDF P32）**：

```c

while (save[i] == k) i += 1;

```

假设：`i→x22, k→x24, save基址→x25`（`save` 为 `long` 数组，元素占 8 字节）

```assembly

Loop:

    slli x10, x22, 3     # x10 = i * 8（因 long 占 8 字节）

    add x10, x10, x25    # x10 = &save[i]

    ld x9, 0(x10)        # x9 = save[i]（RV64 用 ld 加载 8 字节）

    bne x9, x24, Exit    # if save[i] != k → 退出循环

    addi x22, x22, 1     # i = i + 1

    beq x0, x0, Loop     # 无条件跳回 Loop 继续

Exit:

```

  

### 5.4 基本块（Basic Block）

- **定义**：一个基本块是一段连续的指令序列，满足：

  1. 只有第一条指令能被分支指令跳入（**唯一入口**）。

  2. 只有最后一条指令能是分支指令或跳转指令（**唯一出口**）。

- 编译器在基本块层面进行优化（如指令调度、公共子表达式消除）。现代超标量处理器也以基本块为单位进行推测执行。

  
  

## 六、过程调用（函数调用）—— 全书最重难点

  

### 6.1 调用与返回指令

- **`jal rd, label`**（Jump and Link）：将 `PC + 4`（即 `jal` 下一条指令的地址）存入 `rd`，然后跳转到 `label`。通常 `rd = x1`（`ra`）。

  - 例：`jal x1, func` → 调用 `func`，返回地址存入 `x1`。

- **`jalr rd, offset(rs1)`**（Jump and Link Register）：将 `PC + 4` 存入 `rd`，然后跳转到 `rs1 + offset`（12位有符号偏移）。

  - 用于返回：`jalr x0, 0(x1)` → 跳转到 `x1` 处的地址（丢弃返回地址，因为 `rd = x0` 不保存）。

  - 也用于长跳转：`lui` + `jalr` 可实现全 32 位地址跳转。

  

### 6.2 栈帧（Stack Frame）管理

- **栈向低地址增长**：`sp`（`x2`）减小表示分配空间，增大表示回收空间。

- 每个函数在栈上占据一块连续区域，称为**栈帧（Stack Frame）**，用于存放：

  - 返回地址（`ra`）

  - 被调用者保存的寄存器（`s*`）

  - 局部变量（溢出到内存的变量）

  - 传递给被调用函数的参数（若参数超过 `a0~a7` 的 8 个）

  

**被调用者保存（Callee-Saved）**：

- 被调用函数若使用了 `x1`（`ra`）、`x8~x9`、`x18~x27`（`s0~s11`），必须在函数开头将这些寄存器的旧值压栈保存，在函数返回前从栈中恢复。因为调用者期望这些寄存器在调用后保持不变。

  

**调用者保存（Caller-Saved）**：

- 临时寄存器 `x5~x7`、`x28~x31`（`t0~t6`）由调用者负责保存。若调用者在 `jal` 后还需使用这些寄存器的值，必须在自己调用前压栈保存（或在调用后重新计算）。

  

### 6.3 叶过程（Leaf Procedure）示例

**示例（PDF P38~39）**：

```c

int leaf_example(int g, int h, int i, int j) {

    int f = (g + h) - (i + j);

    return f;

}

```

参数映射：`g→x10, h→x11, i→x12, j→x13`，`f→x20`。使用临时寄存器 `x5, x6`。

汇编代码及逐行注释：

```assembly

leaf_example:

    addi sp, sp, -12     # 分配 12 字节栈空间（保存 3 个寄存器）

    sw   x5, 8(sp)       # 保存临时寄存器 x5（调用者可能在使用）

    sw   x6, 4(sp)       # 保存临时寄存器 x6

    sw   x20, 0(sp)      # 保存保存寄存器 x20（必须保存）

    add  x5, x10, x11    # x5 = g + h

    add  x6, x12, x13    # x6 = i + j

    sub  x20, x5, x6     # x20 = (g+h) - (i+j)

    addi x10, x20, 0     # 将返回值放入 a0（x10）

    lw   x20, 0(sp)      # 恢复 x20

    lw   x6, 4(sp)       # 恢复 x6

    lw   x5, 8(sp)       # 恢复 x5

    addi sp, sp, 12      # 回收栈空间

    jalr x0, 0(x1)       # 返回（跳转到 x1 存储的地址）

```

  

### 6.4 非叶过程（递归）示例 —— **最容易出错**

**示例（PDF P43~44）**：

```c

int fact(int n) {

    if (n < 1) return 1;

    else return n * fact(n - 1);

}

```

参数 `n` 在 `x10`，返回值也在 `x10`。这是一个非叶过程（因为它调用了自身），**必须保存 `x1`（返回地址）**，否则递归返回时无法返回调用者。

汇编代码及逐行注释：

```assembly

fact:

    addi sp, sp, -8      # 分配 8 字节栈空间

    sw   x1, 4(sp)       # 保存返回地址（ra）到栈

    sw   x10, 0(sp)      # 保存参数 n（原始值）到栈

    addi x5, x10, -1     # x5 = n - 1

    bge  x5, x0, L1      # if (n - 1 >= 0) 即 n >= 1 → 跳转 L1

    addi x10, x0, 1      # n < 1: 返回值 = 1

    addi sp, sp, 8       # 回收栈空间（无需恢复寄存器）

    jalr x0, 0(x1)       # 返回

L1:

    addi x10, x10, -1    # x10 = n - 1（准备递归参数）

    jal  x1, fact        # 调用 fact(n - 1)，返回后 x10 = fact(n-1)

    addi x6, x10, 0      # x6 = fact(n-1)（保存到保存寄存器）

    lw   x10, 0(sp)      # 恢复原始 n

    lw   x1, 4(sp)       # 恢复返回地址

    addi sp, sp, 8       # 回收栈空间

    mul  x10, x10, x6    # x10 = n * fact(n-1)

    jalr x0, 0(x1)       # 返回

```

  

**易错点**：

1. **忘记保存 `x1`**：在非叶函数中调用 `jal` 会覆盖 `x1`。若不保存，递归返回后 `x1` 丢失，无法返回原始调用者。

2. **忘记保存参数（`x10`）**：递归调用后 `x10` 被破坏，若不保存原始 `n`，则后续乘法无法计算。

3. **栈空间分配与回收不匹配**：每次 `addi sp, sp, -8` 都必须配对 `addi sp, sp, 8`。

  
  

## 七、内存布局与地址空间

  

RISC-V 程序的内存布局（从低地址到高地址）：

  

| 段 | 内容 | 增长方向 |

| :--- | :--- | :--- |

| **文本段（Text）** | 机器指令（只读） | 固定，由链接器确定 |

| **静态数据段（Static Data）** | 全局变量、静态变量、常量字符串 | 固定，`gp`（x3）指向该段中间，便于快速访问 |

| **堆（Heap）** | 动态分配的数据（`malloc` / `new`） | 向**高地址**增长 |

| **（未分配区域）** | 可扩展的虚拟地址空间 | — |

| **栈（Stack）** | 自动变量、栈帧、返回地址 | 向**低地址**增长（`sp` 管理） |

  

**构造 32 位大常数**：

- 当需要加载一个超过 12 位的立即数时（如 `0x12345678`），使用 `lui` + `addi` 组合：

  ```assembly

  lui x10, 0x12345      # x10 = 0x12345000

  addi x10, x10, 0x678  # x10 = 0x12345678

  ```

  
  

## 八、字符与字符串

  

### 8.1 字符编码

- **ASCII**：7 位编码（存于 1 字节中），包含 95 个可打印字符和 33 个控制字符。数字字符 `'0'` 的 ASCII 码为 `0x30`，`'A'` 为 `0x41`，`'a'` 为 `0x61`。

- **Unicode**：32 位字符集，覆盖全球绝大部分文字系统。UTF-8 是变长编码（1~4 字节），向后兼容 ASCII。

  

### 8.2 字符串在 C 语言中

- C 字符串以 **空字符 `\0`（ASCII 码 0x00）** 结尾。

- 字符串不受字节序影响，因为每个字符独立占 1 字节。

  

### 8.3 字符串复制示例（`strcpy`）

**示例（PDF P49~50）**：

```c

void strcpy(char *x, char *y) {

    size_t i = 0;

    while ((x[i] = y[i]) != '\0') i++;

}

```

参数：`x` 在 `x10`，`y` 在 `x11`。

```assembly

strcpy:

    addi sp, sp, -4

    sw   x19, 0(sp)      # 保存 x19（保存寄存器）

    add  x19, x0, x0     # i = 0

Loop:

    add  x5, x19, x11    # x5 = &y[i]

    lbu  x6, 0(x5)       # x6 = y[i]（无符号字节加载，零扩展）

    add  x7, x19, x10    # x7 = &x[i]

    sb   x6, 0(x7)       # x[i] = y[i]（存字节）

    beq  x6, x0, Done    # if y[i] == 0 → 结束循环

    addi x19, x19, 1     # i = i + 1

    beq  x0, x0, Loop    # 无条件跳回

Done:

    lw   x19, 0(sp)      # 恢复 x19

    addi sp, sp, 4

    jalr x0, 0(x1)

```

  
  

## 九、链接与加载

  

### 9.1 编译与汇编流程

1. **预处理**：处理 `#include`、`#define` 等宏指令。

2. **编译**：将 C 源文件翻译为汇编语言（`.s` 或 `.S`）。

3. **汇编**：将汇编语言翻译为**目标文件（Object File，`.o` 或 `.obj`）**，包含机器指令、数据、重定位信息和符号表。

4. **链接（Linking）**：将多个目标文件合并为一个**可执行文件**。链接器完成：

   - **段合并**：将所有目标文件的文本段、数据段分别合并。

   - **符号解析**：确定每个全局符号（函数、全局变量）的地址。

   - **重定位**：修正指令中的地址引用（因为合并后地址发生变化）。

5. **加载（Loading）**：将可执行文件从磁盘复制到内存中，由**加载器（Loader）** 完成：

   - 读取文件头，确定段大小。

   - 创建虚拟地址空间，映射文本段、数据段、栈、堆。

   - 设置 `sp`（栈指针）和 `gp`（全局指针），跳转到启动例程（`_start`），最终调用 `main`。

  

### 9.2 动态链接

- **静态链接**：将所有用到的库函数直接复制到可执行文件中 → 文件体积大，库更新需重新链接。

- **动态链接（共享库/DLL）**：在程序加载时或运行时，将共享库链接到程序中。优点：文件体积小，多个程序共享同一份库代码（内存效率高），库可独立更新。

- **延迟绑定（Lazy Binding）**：首次调用库函数时才解析其地址，加快程序启动速度。

  
  

## 十、常见谬误与陷阱（必读！）

  

1. **陷阱：顺序存储的相邻字，地址相差 4（或 8），而不是 1**。

   - `int` 数组的 `A[0]` 和 `A[1]` 地址相差 4 字节。`lw x5, 4(x10)` 加载的是 `A[1]`，而不是 `A[0]` 的下一个字节。

  

2. **陷阱：忘记保存返回地址 `x1`（非叶函数）**。

   - 任何调用 `jal` 的函数的函数都必须保存 `x1`，否则递归或多层调用会导致返回地址丢失。

  

3. **陷阱：混淆有符号分支（`blt`）与无符号分支（`bltu`）**。

   - 当比较地址或 `size_t` 类型时，必须使用无符号比较（`bltu` / `bgeu`）。使用 `blt` 会导致地址较大（>2GB）时被误判为负数。

  

4. **陷阱：函数返回指向局部变量的指针**。

   - 局部变量在栈帧中，函数返回后栈帧被回收，指针指向的地址可能被其他函数覆盖（悬空指针）。

  

5. **陷阱：`lbu` 与 `lb` 的混淆**。

   - `lb` 会进行**符号扩展**，将 8 位加载值扩展到 XLEN 位。若加载的是字节 `0xFF`，`lb` 存入寄存器为 `0xFFFFFFFF`（-1），而 `lbu` 存入为 `0x000000FF`（255）。

  
  

## 十一、总结性速查表

  

| 类别 | 关键指令 / 概念 | 要点 |

| :--- | :--- | :--- |

| **无条件跳转** | `jal rd, label` | 跳转并保存返回地址到 `rd`（通常 `rd=x1`） |

| **条件跳转** | `beq` / `bne` / `blt` / `bge` | 分支偏移以 2 字节为单位（PC + imm × 2） |

| **内存加载** | `lw` / `lh` / `lb` / `ld` | 带符号扩展（`lbu` / `lhu` 为无符号/零扩展） |

| **内存存储** | `sw` / `sh` / `sb` / `sd` | 只存储寄存器低端对应字节数 |

| **栈操作** | `addi sp, sp, -N` / `addi sp, sp, N` | 栈向低地址增长，分配/回收对齐（通常 8/16 字节） |

| **被调用者保存** | `x1`（ra）, `x8~x9`, `x18~x27`（s0~s11） | 被调用函数若使用必须压栈保存 |

| **调用者保存** | `x5~x7`, `x28~x31`（t0~t6） | 调用者若需保留必须自己压栈 |

| **参数/返回值** | `x10~x17`（a0~a7） | 参数传递；返回值放 `x10` |

# 第4章：处理器体系结构

  

## 一、CPU性能公式与设计基础

  

### 1.1 CPU性能三要素

CPU执行一个程序的总时间由三个因素共同决定：

$$\text{CPU时间} = \text{指令数} \times \text{CPI} \times \text{时钟周期}$$

  

| 因素 | 定义 | 主要决定者 |

| :--- | :--- | :--- |

| **指令数（IC，Instruction Count）** | 程序执行的总指令条数 | ISA（指令集）和编译器优化 |

| **CPI（Cycles Per Instruction）** | 每条指令平均消耗的时钟周期数 | CPU微架构（流水线、超标量等） |

| **时钟周期（Clock Period）** | 一个时钟脉冲的持续时间（如1ns对应1GHz） | 硬件实现技术（工艺节点、关键路径延迟） |

  

### 1.2 指令执行的五个阶段（所有指令的通用框架）

任何指令的执行都可以分解为以下五个阶段（不同指令可能省略其中若干阶段）：

  

| 阶段 | 英文 | 操作 | 涉及的硬件 |

| :--- | :--- | :--- | :--- |

| **IF** | Instruction Fetch | 从指令存储器读取指令，PC更新 | 指令存储器、PC、加法器（PC+4） |

| **ID** | Instruction Decode / Register Read | 译码指令，从寄存器堆读取源操作数 | 控制器、寄存器堆 |

| **EX** | Execute | ALU执行运算（算术结果、地址计算、分支比较） | ALU、符号扩展单元 |

| **MEM** | Memory Access | 读/写数据存储器（仅`lw`/`sw`需要） | 数据存储器 |

| **WB** | Write Back | 将结果写回寄存器堆（`lw`和R型指令需要） | 寄存器堆 |

  
  

## 二、逻辑设计基础

  

在深入处理器设计之前，必须理解数字逻辑电路的两大类基本元件：

  

### 2.1 组合逻辑元件（Combinational Elements）

- **定义**：输出仅取决于当前输入，与历史状态无关（无存储功能）。

- **常见元件**：

  - **与门（AND Gate）**：`Y = A & B`

  - **多路选择器（MUX）**：`Y = S ? I1 : I0`，根据选择信号`S`从两路输入中选一路。

  - **加法器（Adder）**：`Y = A + B`

  - **ALU（算术逻辑单元）**：`Y = F(A, B)`，功能由控制信号决定（加、减、与、或、异或等）。

  

### 2.2 时序逻辑元件（Sequential Elements）

- **定义**：输出取决于当前输入和当前存储的状态（有存储功能）。它们由**时钟信号（Clock）** 驱动。

- **边沿触发（Edge-Triggered）**：寄存器仅在时钟的**上升沿（0→1）** 更新存储的值。这保证了在同一个时钟周期内，组合逻辑的输出稳定后才会被锁存到寄存器中。

- **带写使能的寄存器**：除了时钟边沿外，还需要`Write Enable`（写使能）信号为1时才更新。若写使能为0，寄存器保持原值不变。这在寄存器堆和带控制信号的流水线寄存器中广泛使用。

  

### 2.3 时钟周期与关键路径

- 组合逻辑的延迟决定了**关键路径（Critical Path）**——即所有指令中最长的组合逻辑路径。

- **时钟周期必须 ≥ 关键路径延迟 + 寄存器开销**（寄存器建立时间+时钟歪斜等）。

- 时钟周期越短，CPU频率越高，但关键路径限制了时钟周期的下限。

  
  

## 三、单周期数据通路与控制（4.2 ~ 4.4）

  

### 3.1 单周期CPU的基本概念

- **定义**：所有指令在**一个时钟周期内**完成全部五个阶段（IF、ID、EX、MEM、WB）。

- **优点**：控制逻辑简单，每条指令的CPI = 1。

- **缺点**：时钟周期必须满足**最慢指令（`lw`）** 的延迟要求，导致时钟周期很长，整体性能低下。

  

### 3.2 数据通路的逐步构建

  

#### 3.2.1 取指（IF）阶段

- **PC**（程序计数器）：存储当前指令地址。

- **指令存储器**：以PC为地址读取32位指令。

- **PC更新**：正常情况下 `PC ← PC + 4`（RISC-V指令为32位=4字节）。

  

#### 3.2.2 R型指令的数据通路（`add`, `sub`, `and`, `or`, `xor`）

- **ID阶段**：从寄存器堆读取 `rs1` 和 `rs2` 的值。

- **EX阶段**：ALU根据 `funct3` 和 `funct7` 对两个源操作数执行运算。

- **WB阶段**：将ALU结果写回寄存器堆的 `rd`。

- **所需硬件**：寄存器堆（2读1写）、ALU。

  

#### 3.2.3 加载/存储指令的数据通路（`lw`, `sw`）

- **ID阶段**：从寄存器堆读取基址寄存器（`rs1`）的值。

- **EX阶段**：ALU计算 **内存地址 = 基址 + 符号扩展的立即数**。

- **MEM阶段**（关键区别）：

  - `lw`：从数据存储器读出数据 → WB阶段写回寄存器堆。

  - `sw`：将 `rs2` 的值写入数据存储器（不写回寄存器堆）。

- **所需额外硬件**：符号扩展单元、数据存储器。

  

#### 3.2.4 分支指令的数据通路（`beq`, `bne`）

- **ID阶段**：读取 `rs1` 和 `rs2`。

- **EX阶段**：ALU执行**减法**，产生 `Zero` 信号（若差为0则 `Zero=1`）。

- **同时**：计算**分支目标地址 = PC + 符号扩展的立即数 × 2**（RISC-V分支偏移以2字节为单位）。

- **控制决策**：若 `Zero=1`（对于`beq`）或 `Zero=0`（对于`bne`），则 `PC ← 分支目标地址`；否则 `PC ← PC + 4`。

  

### 3.3 控制信号表（核心，必须理解每个信号的作用）

  

| 控制信号 | 作用 | R型 | `lw` | `sw` | `beq` |

| :--- | :--- | :--- | :--- | :--- | :--- |

| **RegWrite** | 是否写回寄存器堆 | 1（写） | 1（写） | 0（不写） | 0（不写） |

| **ALUSrc** | ALU第二操作数来源：0=寄存器，1=立即数 | 0 | 1 | 1 | 0 |

| **MemRead** | 是否读数据存储器 | 0 | 1 | 0 | 0 |

| **MemWrite** | 是否写数据存储器 | 0 | 0 | 1 | 0 |

| **MemtoReg** | 写回数据来源：0=ALU结果，1=内存读数据 | 0 | 1 | X（无效） | X（无效） |

| **ALUOp（2位）** | ALU操作类型（与funct字段配合生成精确控制） | 10 | 00 | 00 | 01 |

  

### 3.4 ALU控制单元

- ALUOp（2位）由主控制器根据指令类型生成，然后与 `funct3` 和 `funct7` 字段组合，生成ALU的精确控制信号（4位）。

  

| ALUOp | 指令类型 | funct字段 | ALU操作 | ALU控制 |

| :--- | :--- | :--- | :--- | :--- |

| 00 | `lw` / `sw` | X | 加法 | 0010 |

| 01 | `beq` / `bne` | X | 减法 | 0110 |

| 10 | R型 | 0000000 + funct3=000 | 加法 | 0010 |

| 10 | R型 | 0100000 + funct3=000 | 减法 | 0110 |

| 10 | R型 | 0000000 + funct3=111 | 与 | 0000 |

| 10 | R型 | 0000000 + funct3=110 | 或 | 0001 |

  

### 3.5 关键路径与性能瓶颈

- **关键路径**：**`lw`（加载）指令**的路径最长。

  - IF（取指）→ Regs（读寄存器）→ ALU（计算地址）→ D-Mem（读数据）→ Regs（写回）

  - 这条路径包含了最多最慢的组件（特别是数据存储器访问），决定了单周期CPU的时钟周期。

- 单周期CPU的CPI = 1，但时钟周期很长，导致整体性能远低于流水线CPU。

  
  

## 四、流水线基础（4.5）

  

### 4.1 流水线的核心思想：提高吞吐率，而非降低延迟

- **吞吐率（Throughput）**：单位时间内完成的指令数。流水线通过并行执行多条指令来大幅提高吞吐率。

- **延迟（Latency）**：单条指令从取指到写回的总时间。流水线**不会减少**单条指令的延迟（甚至可能因流水线寄存器开销略有增加）。

- **理想加速比** = 流水级数。例如5级流水线，理想情况下吞吐率提高5倍。

  

### 4.2 洗衣房类比

- 单周期（串行）：4批衣服，每批需2小时（洗1h + 烘1h）→ 总时间8小时。

- 流水线：洗第1批 → 洗第2批（同时烘第1批）→ ... → 总时间3.5小时。

- 加速比 = 8 / 3.5 ≈ 2.3倍（理想4倍，因阶段不平衡略有损失）。

  

### 4.3 RISC-V对流水线的友好设计（PDF P34）

| 设计特性 | 对流水线的好处 |

| :--- | :--- |

| 所有指令32位定长 | 取指译码简单，IF阶段可在一个周期内完成 |

| 指令格式规整（rs1/rs2位置固定） | ID阶段可同时译码和读取寄存器堆 |

| 仅`lw`/`sw`访存，且地址在EX计算 | MEM阶段专用于访存，流水线划分清晰 |

| 无复杂寻址模式（无内存-内存操作） | 避免了多周期内存操作导致的流水线停顿 |

  

### 4.4 流水线寄存器（Pipeline Registers）

- 为了在相邻阶段之间传递数据和控制信息，需要在每个阶段边界插入**流水线寄存器**。

- 5个阶段对应四个流水线寄存器：

  - **IF/ID**：保存取出的指令和PC+4。

  - **ID/EX**：保存译码后的控制信号、读出的寄存器值、立即数、PC等。

  - **EX/MEM**：保存ALU结果、分支目标、控制信号等。

  - **MEM/WB**：保存内存读出的数据、ALU结果、控制信号等。

  

### 4.5 性能对比（PDF P31）

- 单周期：`lw`需800ps，`sw`需700ps，R型需600ps，`beq`需500ps。时钟周期必须≥800ps。

- 流水线：每阶段平衡为200ps（加上寄存器开销约20ps），时钟周期≈220ps。

- 虽然`lw`在流水线中仍需800ps的“原始时间”，但每个周期可完成一条新指令，吞吐率大幅提升。

  
  

## 五、冒险（Hazards）—— 全书核心重难点

  

冒险是指流水线中由于指令间的依赖关系，导致无法在下一个时钟周期继续执行后续指令的情况。共分三类：**结构冒险**、**数据冒险**、**控制冒险**。

  

### 5.1 结构冒险（Structural Hazard）

- **定义**：硬件资源不足，导致多条指令在同一时钟周期争抢同一硬件单元。

- **经典场景**：指令存储器和数据存储器共用同一物理内存（冯·诺依曼结构）。

  - 第1条指令在MEM阶段访问数据存储器，第2条指令在IF阶段取指，冲突。

- **RISC-V解决方案**：采用**分离的指令Cache和数据Cache**（哈佛结构变体），或使用独立的指令存储器 and 数据存储器。

  

### 5.2 数据冒险（Data Hazard）—— 最核心考点

- **定义**：后续指令需要前序指令的计算结果，但该结果尚未产生。

  

**典型场景**：

```assembly

sub x2, x1, x3   # x2 在 EX 阶段末尾才产生

and x12, x2, x5  # and 在 EX 阶段需要 x2，但 sub 的结果尚未写回

```

  

#### 5.2.1 解决方案一：转发（Forwarding / Bypassing）—— 硬件核心机制

- **思想**：当ALU计算结果在EX/MEM或MEM/WB流水线寄存器中已经存在时，直接通过**旁路路径**将其送到当前指令的ALU输入端，而无需等待写回寄存器堆。

- **转发路径**：

  - **EX/MEM → EX**：前一条指令的ALU结果转发给当前指令（最常见）。

  - **MEM/WB → EX**：前两条指令的结果（可能是ALU结果或Load读出的数据）转发给当前指令。

  

- **转发条件检测（必须掌握）**：

  在ID/EX阶段的指令，其源寄存器为 `ID/EX.RegisterRs1` 和 `ID/EX.RegisterRs2`。我们需要检测前序指令是否将结果写入这些寄存器。

  

  **EX冒险（ForwardA = 10）**：

  如果前一条指令（位于EX/MEM阶段）将写回寄存器，且目标寄存器与当前源寄存器相同，则转发。

  ```

  if (EX/MEM.RegWrite && EX/MEM.Rd != 0 && EX/MEM.Rd == ID/EX.Rs1)

      ForwardA = 10;  // 从EX/MEM转发到ALU输入A

  if (EX/MEM.RegWrite && EX/MEM.Rd != 0 && EX/MEM.Rd == ID/EX.Rs2)

      ForwardB = 10;  // 从EX/MEM转发到ALU输入B

  ```

  

  **MEM冒险（ForwardA = 01）**：

  如果前两条指令（位于MEM/WB阶段）将写回寄存器，且目标寄存器与当前源寄存器相同，则转发。

  ```

  if (MEM/WB.RegWrite && MEM/WB.Rd != 0 && MEM/WB.Rd == ID/EX.Rs1)

      ForwardA = 01;  // 从MEM/WB转发

  if (MEM/WB.RegWrite && MEM/WB.Rd != 0 && MEM/WB.Rd == ID/EX.Rs2)

      ForwardB = 01;  // 从MEM/WB转发

  ```

  

  **双数据冒险（Double Data Hazard）—— 优先转发最新值**：

  考虑序列：

  ```assembly

  add x1, x1, x2  # I1

  add x1, x1, x3  # I2

  add x1, x1, x4  # I3

  ```

  对于I3，`x1` 同时满足EX冒险（来自I2）和MEM冒险（来自I1）。必须**优先选择EX/MEM的最新结果**。因此在MEM冒险条件中要加上 `not(EX/MEM条件)`：

  ```

  if (MEM/WB.RegWrite && MEM/WB.Rd != 0 && MEM/WB.Rd == ID/EX.Rs1

      && !(EX/MEM.RegWrite && EX/MEM.Rd != 0 && EX/MEM.Rd == ID/EX.Rs1))

      ForwardA = 01;

  ```

  

  **易错点**：必须检查 `Rd != 0`，因为写入 `x0` 是丢弃结果，不应转发。

  

#### 5.2.2 解决方案二：停顿（Stall / Bubble）—— 转发无法解决的情况

- **Load-Use冒险**：

  ```assembly

  lw x10, 0(x1)    # I1：x10 在 MEM 阶段末尾才从内存读出

  add x11, x10, x2 # I2：add 的 EX 阶段在 I1 的 MEM 阶段之前开始，无法转发

  ```

  即使有转发，`add` 的EX开始时，`lw` 的数据还没有从内存读出，必须**停顿（Stall）一个周期**。

  

- **停顿检测条件**：

  ```

  if (ID/EX.MemRead == 1                    // 前一条指令是 Load

      && (ID/EX.Rd == IF/ID.Rs1             // 加载的目标寄存器与当前指令的 rs1 相同

          || ID/EX.Rd == IF/ID.Rs2))        // 或与 rs2 相同

      stall = 1;

  ```

  

- **停顿操作（插入空泡）**：

  1. **冻结 PC 和 IF/ID 寄存器**：不更新，保持当前取指指令不变。

  2. **将 ID/EX 控制信号全部清零**：使得下一个周期进入 EX 阶段的是一条空泡（NOP），不执行任何有效操作。

  3. 一个周期后，`lw` 的数据从 MEM/WB 读出，可通过转发送到 `add` 的 EX 阶段。

  

### 5.3 控制冒险（Control Hazard）—— 分支指令

- **定义**：分支指令的跳转方向（是否跳转）和跳转目标在取指阶段无法确定，导致取指可能取错。

- **分支惩罚（Branch Penalty）**：从取指到确定分支结果所需的时间。

  

#### 5.3.1 尽早判断（缩短分支惩罚）

- **改进前**：分支比较在EX阶段完成，目标地址也在EX计算 → 若预测错误，已取出2条错误指令（IF和ID），惩罚为2个周期。

- **改进后（RISC-V实现）**：将分支比较和地址计算**提前到ID阶段**。

  - 在ID阶段增加专用**比较器**和**加法器**。

  - 分支目标在ID结束时已知 → 若预测错误，仅1条指令（已在IF取指）需要冲刷，惩罚降为1个周期。

  

#### 5.3.2 静态分支预测

- **RISC-V默认策略**：**预测分支不跳转（Predict Not Taken）**。

- 若预测正确（不跳转）→ 无惩罚，继续顺序取指。

- 若预测错误（实际跳转）→ 冲刷（Flush）已取指的指令（将IF/ID和ID/EX中的指令清空为NOP），PC指向分支目标。

  

#### 5.3.3 动态分支预测（更深流水线）

当流水线级数更深（如12级、20级），分支惩罚更大，静态预测不够，需要动态预测。

  

- **1位预测器（BHT）**：

  - 记录每个分支上次是否跳转（1位状态：Taken / Not Taken）。

  - **缺点**：在循环中表现差。内层循环的最后一次迭代预测错误（退出循环），下一次循环开始又预测错误（进入循环）。连续两次错误。

  

- **2位预测器**：

  - 有4个状态（强跳转、弱跳转、弱不跳转、强不跳转）。

  - 只有**连续两次**预测错误才翻转预测方向。

  - 大大降低了循环分支的预测错误率。

  

- **分支目标缓冲器（BTB，Branch Target Buffer）**：

  - 缓存分支指令的目标地址。

  - 取指时，若PC命中BTB且预测为Taken，可直接从目标地址取指，无需等待地址计算。

  

#### 5.3.4 分支惩罚计算示例

若分支在ID阶段判断，预测错误时已有1条指令在IF阶段，需冲刷（清空IF/ID），惩罚=1个周期。在更深的流水线（如Pentium 4的31级），惩罚可能高达20个周期，动态预测至关重要。

  
  

## 六、异常（Exceptions）与中断（Interrupts）（4.9）

  

### 6.1 异常 vs 中断

| 类型 | 来源 | 示例 |

| :--- | :--- | :--- |

| **异常（Exception）** | CPU内部事件 | 未定义指令（illegal opcode）、系统调用（ecall）、硬件故障、算术溢出 |

| **中断（Interrupt）** | 外部I/O设备 | 键盘输入、硬盘DMA完成、定时器超时 |

  

### 6.2 精确异常（Precise Exception）—— 流水线必须保证

- **定义**：异常发生时，处理器必须满足：

  1. 异常指令之前的所有指令**完整执行完毕**。

  2. 异常指令及其之后的所有指令**被冲刷（Flush）**，不产生副作用。

  3. 异常处理程序返回时，从异常指令重新开始执行（若可恢复）。

  

- **RISC-V实现**：

  - 保存异常PC到 **SEPC**（Supervisor Exception Program Counter）。

  - 保存异常原因到 **SCAUSE**（Supervisor Exception Cause）。

  - 跳转到异常处理程序入口（通常为 `0x1C090000` 或由`stvec`指定）。

  

### 6.3 流水线中的异常处理

- **示例（PDF P93~95）**：

  ```assembly

  40: sub x11, x2, x4

  44: and x12, x2, x5

  48: or  x13, x2, x6

  4c: add x1, x2, x1   # 此指令发生硬件故障

  50: sub x15, x6, x7

  54: lw  x16, 100(x7)

  ```

  当 `add` 在EX阶段发生异常时：

  1. `sub`、`and`、`or` 已完成（或即将完成）→ 允许它们写完（或已写完）结果。

  2. `add` 及其之后的 `sub`、`lw` 被冲刷（清空）。

  3. SEPC = `0x4c`（异常指令地址），SCAUSE = 异常码。

  4. PC ← 处理程序入口。

  5. 处理程序返回后，从 `0x4c` 重新执行 `add`。

  

- **多异常同时发生**：

  - 多条指令在不同阶段可能同时产生异常（如一条在MEM数据访问异常，另一条在EX未定义指令）。

  - **原则**：优先处理**最早（程序顺序最靠前）** 的异常。冲刷后续所有指令。

  

- **难点**：复杂流水线（超标量、乱序执行）中维护精确异常非常困难，常采用**不精确异常（Imprecise Exception）**，由软件处理程序自行恢复。

  
  

## 七、指令级并行（ILP）与多发射（4.10 ~ 4.12）

  

### 7.1 多发射（Multiple Issue）的基本概念

- **单发射（Single Issue）**：每个时钟周期最多发射1条指令（CPI $\ge$ 1）。

- **多发射（Multiple Issue）**：每个时钟周期发射多条指令（CPI < 1，使用IPC：Instructions Per Cycle来衡量）。

- **分类**：

  1. **静态多发射**：编译器将指令打包成“发射包”，硬件并行执行。

  2. **动态多发射**：硬件在运行时动态决定发射哪些指令。

  

### 7.2 静态多发射（VLIW——Very Long Instruction Word）

- **思想**：编译器预先分析指令间的依赖关系，将无依赖的指令合并到一个**超长指令字**中，包含多个独立的操作。

- **关键技术**：

  - **循环展开（Loop Unrolling）**：复制循环体多次，减少循环开销，暴露更多并行机会。

  - **寄存器重命名（Register Renaming）**：为每次循环迭代分配不同的物理寄存器，消除“反依赖”（名称依赖），避免不必要的伪冲突。

- **双发射RISC-V调度示例（PDF P108）**：

  ```assembly

  Loop: lw  x31, 0(x20)   # Load

        add x31, x31, x21

        sw  x31, 0(x20)

        addi x20, x20, -4

        blt x22, x20, Loop

  ```

  手动调度为双发射（ALU/分支槽 + Load/Store槽）：

  | 周期 | ALU/分支槽 | Load/Store槽 |

  | :--- | :--- | :--- |

  | 1 | NOP | `lw x31, 0(x20)` |

  | 2 | `addi x20, x20, -4` | NOP（`lw`结果未就绪） |

  | 3 | `add x31, x31, x21` | NOP（等待ALU） |

  | 4 | `blt x22, x20, Loop` | `sw x31, 4(x20)`（注意偏移调整） |

  - IPC = 5条指令 / 4周期 = 1.25（小于峰值2）。

- **循环展开后的调度（PDF P110）**：

  展开4次循环体，IPC提升至 14/8 = 1.75，更接近峰值2，但代价是代码膨胀和寄存器消耗增加。

  

### 7.3 动态多发射（超标量 Superscalar）

- **思想**：由CPU硬件（而非编译器）在运行时动态决定每个周期发射几条指令。

- **核心优点**：能处理编译器无法预见的运行时情况（如Cache Miss）。

- **关键技术**：

  1. **乱序执行（Out-of-Order Execution）**：

     - 指令不按程序顺序执行，但**顺序提交（In-Order Commit）** 以保证精确异常。

     - 使用**保留站（Reservation Stations）**：为每条等待执行的指令缓存操作数和功能单元信息。

     - 使用**重排序缓冲（ROB，Reorder Buffer）**：保存已执行但尚未提交的结果，并按程序顺序提交。

  2. **寄存器重命名（Register Renaming）**：

     - 将体系结构寄存器（x0~x31）动态映射到一组更大的物理寄存器。

     - 消除WAR（写后读）和WAW（写后写）伪依赖，仅保留真依赖（RAW，读后写）。

  3. **推测执行（Speculation）**：

     - **分支推测**：预测分支方向，提前执行推测路径上的指令。若预测错误，冲刷ROB，丢弃结果。

     - **加载推测**：提前执行Load指令，绕过可能未完成的Store指令。若Store地址与Load地址冲突，则重新执行Load。

  

- **难点**：动态调度的硬件复杂度极高，功耗巨大（见PDF P118表格：Pentium 4功耗103W vs i486的5W）。多核+简单流水线成为能效导向的首选。

  
  

## 八、真实处理器对比（4.13 节选）

  

| 处理器 | 工艺/市场 | 流水线级数 | 发射宽度 | 调度策略 | 功耗/频率 |

| :--- | :--- | :--- | :--- | :--- | :--- |

| **ARM Cortex-A53** | 移动设备 | 8级 | 2-issue | 静态有序（In-Order） | ~100mW @ 1GHz |

| **Intel Core i7 920** | 服务器/桌面 | 14级 | 4-issue | 动态乱序（Out-of-Order） | 130W @ 2.66GHz |

- **结论**：高性能（Core i7）依赖深流水线+乱序执行，但功耗极高；移动（Cortex-A53）采用较简单有序流水线，追求能效比。

  
  

## 九、常见谬误与陷阱（必读！）

  

1. **谬误：流水线降低单条指令的执行时间（延迟）**。

   - 流水线提高吞吐率，但不减少延迟。单条`lw`在流水线中仍需要IF+ID+EX+MEM+WB五个阶段的总时间（加上寄存器开销），和单周期一样长。

  

2. **陷阱：转发时忘记排除 `x0`**。

   - 转发条件必须加上 `RegWrite` 且目标寄存器不为 `x0`（即 `Rd != 0`），否则会将写入`x0`的“结果”转发给后续指令，破坏程序语义（写入x0应丢弃结果）。

  

3. **陷阱：双数据冒险优先级错误**。

   - 当同时满足EX冒险和MEM冒险时，必须选择EX/MEM（最新结果），否则会使用过期的旧值。条件判断中要用 `not(EX/MEM条件)` 屏蔽MEM转发。

  

4. **陷阱：分支预测错误后忘记冲刷流水线**。

   - 若预测不跳转但实际跳转，必须将已预取的指令（在IF/ID和ID/EX中）清空为NOP，否则会错误执行分支延迟槽后的指令。

  

5. **陷阱：忽略异常指令之后的指令**。

   - 异常发生时，必须冲刷异常指令及其之后的所有指令，并丢弃它们的中间结果，否则会造成不可恢复的状态污染。

  

6. **难点：复杂流水线中CPI不完全等于 $1/\text{发射宽度}$**。

   - 冒险、Cache Miss、分支错误预测等导致流水线频繁停顿，实际CPI远低于理想值。性能分析必须结合真实程序行为模拟。

  
  

## 十、总结性速查表

  

| 概念 | 关键公式 / 机制 | 核心要点 |

| :--- | :--- | :--- |

| CPU时间 | $\text{IC} \times \text{CPI} \times \text{周期}$ | 三要素共同决定性能 |

| 单周期 | 所有指令1周期完成 | CPI=1，但周期长（受`lw`限制） |

| 流水线 | 5级（IF, ID, EX, MEM, WB） | 提高吞吐率，不降延迟，理想加速比=5 |

| 转发（前递） | EX/MEM→EX, MEM/WB→EX | 解决大多数ALU数据冒险，需检查`Rd`和`Rs` |

| 停顿（气泡） | 冻结PC和IF/ID，清空ID/EX | 解决Load-Use冒险 |

| 分支预测（静态） | 预测不跳转（RISC-V默认） | 预测错误惩罚1个周期（ID阶段判断） |

| 动态预测 | 2位预测器 + BTB | 降低深层流水线的分支惩罚 |

| 精确异常 | SEPC + SCAUSE + 冲刷 | 异常前指令完成，异常后指令清空 |

| VLIW（静态多发射） | 编译器打包无依赖指令 | 需要循环展开和寄存器重命名 |

| 超标量（动态多发射） | 乱序执行 + ROB + 保留站 | 硬件复杂，功耗高，但适应性强 |

# 第6章：存储器层次结构

  
  

## 一、局部性原理（Principle of Locality）—— 存储器层次结构的理论基础

  

### 1.1 局部性的两种形式

程序在任何时刻都倾向于访问其地址空间的一个较小部分，这种现象被称为**局部性**。

  

| 类型 | 定义 | 典型示例 |

| :--- | :--- | :--- |

| **时间局部性（Temporal Locality）** | 刚刚访问过的数据/指令很可能在不久的将来再次被访问 | 循环体中的代码、循环计数器（`i++`）、频繁调用的函数 |

| **空间局部性（Spatial Locality）** | 刚刚访问过的地址附近的数据/指令很可能很快被访问 | 顺序执行的指令、数组的顺序遍历、结构体中的连续字段 |

  

### 1.2 局部性为何有效

- **指令访问**：程序顺序执行（空间局部性），循环反复执行（时间局部性）。

- **数据访问**：数组遍历（空间局部性），频繁使用同一变量（时间局部性）。

- **量化证据**：SPEC基准测试表明，程序90%以上的执行时间花费在10%以下的代码中（90/10定律）。

  
  

## 二、存储器层次结构概述

  

### 2.1 层次结构金字塔（从上到下，速度递减、容量递增、成本递减）

| 层级 | 技术 | 典型访问时间 | 典型容量 | 管理方 |

| :--- | :--- | :--- | :--- | :--- |

| **寄存器** | SRAM/触发器 | < 1 ns | 几百字节 | 编译器 |

| **L1 Cache（指令/数据）** | SRAM | 1 ~ 4 个时钟周期 | 16 ~ 64 KB | 硬件 |

| **L2 Cache** | SRAM | 10 ~ 20 个时钟周期 | 128 KB ~ 数 MB | 硬件 |

| **L3 Cache** | SRAM | 30 ~ 50 个时钟周期 | 数 MB ~ 数十 MB | 硬件 |

| **主存（Main Memory）** | DRAM | 50 ~ 100 ns（~200-400周期） | 数 GB ~ 数百 GB | 硬件+OS |

| **本地二级存储（SSD）** | NAND Flash | 10 ~ 100 μs | 数百 GB ~ 数 TB | OS |

| **远程二级存储（HDD）** | 磁盘 | 5 ~ 20 ms | 数 TB ~ 数十 TB | OS/管理员 |

  

### 2.2 块（Block / Line）—— 数据传输的基本单元

- **定义**：在相邻两层存储器之间传输数据的最小单位。

- 例如：Cache与主存之间以**Cache块（通常64字节）** 为单位传输；主存与磁盘之间以**页（通常4KB）** 为单位传输。

  

### 2.3 命中与缺失（Hit / Miss）

- **命中（Hit）**：CPU请求的数据在高层存储器中找到。

  - **命中率（Hit Rate）** = 命中次数 / 总访问次数。

- **缺失（Miss）**：CPU请求的数据不在高层存储器中，必须从低层复制。

  - **缺失率（Miss Rate）** = 1 - 命中率。

  - **缺失惩罚（Miss Penalty）**：从低层向高层传输一个块所需的时间（对性能影响极大）。

  
  

## 三、存储技术详解（6.2）

  

### 3.1 SRAM（静态随机存取存储器）

- **技术原理**：使用**双稳态触发器（6个晶体管/bit）** 存储数据。

- **特点**：

  - 无需刷新（保持数据只要不断电）

  - 速度极快（0.5 ~ 2.5 ns）

  - 价格昂贵（$2000 ~ $5000/GB）

  - 集成度低 → 容量小

- **用途**：Cache（高速缓存）

  

### 3.2 DRAM（动态随机存取存储器）

- **技术原理**：使用**电容（1个晶体管+1个电容/bit）** 存储电荷。

- **特点**：

  - 电容会缓慢放电，必须**周期性刷新（Refresh）**（约每64ms一次）

  - 速度较快（50 ~ 70 ns）

  - 价格便宜（$20 ~ $75/GB）

  - 集成度高 → 容量大

- **用途**：主存

- **DRAM行缓冲（Row Buffer）**：

  - 内部组织为二维矩阵（行×列）。

  - 访问一行时，整行数据被读入行缓冲器，同一行的后续访问（突发模式）延迟大大降低。

  

### 3.3 DRAM性能增强技术

| 技术 | 原理 | 效果 |

| :--- | :--- | :--- |

| **SDRAM（同步DRAM）** | 与系统时钟同步，支持突发传输 | 无需逐列发送地址，连续数据流式输出 |

| **DDR（双倍数据率）** | 时钟上升沿和下降沿各传一次数据 | 带宽翻倍 |

| **QDR（四倍数据率）** | 输入和输出各有独立的DDR通道 | 带宽再翻倍 |

| **多通道（Banking）** | 多个DRAM芯片并行工作 | 带宽随通道数线性增加 |

| **宽总线（Wide Bus）** | 数据总线宽度从1字扩展到多字 | 单次传输更多数据 |

  

### 3.4 磁盘（Hard Disk Drive）—— 访问时间计算（必考）

- **组织结构**：盘片（Platters）→ 磁道（Tracks）→ 扇区（Sectors）。

- **访问时间的四个组成部分**：

  $$\text{磁盘访问时间} = \text{寻道时间} + \text{旋转延迟} + \text{传输时间} + \text{控制器开销}$$

  

| 组成部分 | 定义 | 典型值 |

| :--- | :--- | :--- |

| **寻道时间（Seek Time）** | 磁头移动到目标磁道所需时间 | 2 ~ 10 ms（平均） |

| **旋转延迟（Rotational Latency）** | 磁盘旋转到目标扇区所需时间 | 平均为半圈时间 |

| **传输时间（Transfer Time）** | 数据从磁盘读出/写入的时间 | 取决于传输速率和块大小 |

| **控制器开销** | 控制器处理命令和校验的时间 | < 0.5 ms |

  

- **完整计算例题（PDF P15）**：

  给定参数：512字节扇区、15000 RPM（每分钟转数）、平均寻道时间4ms、传输率100MB/s、控制器开销0.2ms。

  

  **Step 1**：计算旋转延迟。

  - $1\text{转的时间} = 60 / 15000 = 0.004\text{秒} = 4\text{ms}$。

  - $\text{平均旋转延迟} = \text{半圈} = 4\text{ms} / 2 = 2\text{ms}$。

  

  **Step 2**：计算传输时间。

  - 512字节 = 0.5 KB = 0.0005 MB（注意单位统一）。

  - $\text{传输时间} = 0.0005 \text{ MB} / 100 \text{ MB/s} = 0.000005\text{秒} = 0.005\text{ms}$。

  

  **Step 3**：汇总。

  - $\text{平均读时间} = 4\text{ms} + 2\text{ms} + 0.005\text{ms} + 0.2\text{ms} = 6.205\text{ms} \approx 6.2\text{ms}$。

  

  **易错点**：

  - 转速15000 RPM → 每转4ms，不是每秒4ms。

  - 单位换算：512B = 0.5KB = 0.0005MB（除以1024两次），或直接用字节计算：512B / 100MB/s = 512 / 100,000,000 = 0.00000512s = 0.00512ms。

  

### 3.5 闪存（Flash Memory）

- **特点**：

  - 非易失性（断电不丢数据）

  - 速度比磁盘快100~1000倍

  - 功耗低、抗震性强

  - 价格介于DRAM和磁盘之间

- **类型**：

  - **NOR Flash**：随机读/写，用于嵌入式系统（如BIOS、手机固件）

  - **NAND Flash**：块访问（类似磁盘），容量大、便宜，用于SSD、U盘、SD卡

- **重要缺点**：每个块的擦写次数有限（约 $10^4 \sim 10^5$ 次），需要**磨损均衡（Wear Leveling）** 算法延长寿命。

  
  

## 四、Cache基础：直接映射（6.3）

  

### 4.1 Cache的基本结构

- Cache是一个小而快的存储器，位于CPU与主存之间，存放最近访问的数据/指令的副本。

- **容量**：通常为16KB ~ 数MB。

- **块大小（Block Size）**：Cache与主存之间的数据传输单位，通常为16 ~ 128字节（如4~8个字）。

  

### 4.2 直接映射（Direct Mapped）

- **定义**：每个主存块只能放到Cache中**唯一固定**的位置。

- **位置计算公式**：

  索引（Index）= 块地址（Block Address）对 Cache 块数（#Blocks）取模

  其中：$$\text{块地址} = \left\lfloor \frac{\text{内存地址}}{\text{块大小}} \right\rfloor$$

- **完整例题（PDF P27）**：

  给定：64块Cache，每块16字节。问：地址1200映射到哪个Cache块？

  

  **Step 1**：计算块地址。

  - 块地址 = 1200 // 16 = **75**（整除）。

  

  **Step 2**：计算索引。

  - 索引 = 75 % 64 = **11**。

  

  答案：块号11。

  

### 4.3 地址划分（3个字段）

以一个32位地址、块大小16字节（$2^4$）、Cache容量16KB（有1024块=$2^{10}$）为例：

  

| 字段 | 位数 | 作用 | 计算 |

| :--- | :--- | :--- | :--- |

| **偏移（Offset）** | 4位 | 在块内定位字节 | $\log_2(\text{块大小})$ |

| **索引（Index）** | 10位 | 定位到Cache中的某一行 | $\log_2(\text{块数})$ |

| **标记（Tag）** | 18位 | 与存储的Tag比较，判断是否命中 | $32 - \text{Offset} - \text{Index}$ |

  

- **标记（Tag）**：存储主存地址的高位部分。当CPU访问时，提取地址的Tag字段，与Cache中对应索引位置的Tag比较，若相等且Valid位为1，则命中。

  

### 4.4 Valid位与初始状态

- **Valid位（有效位）**：1表示该Cache行存放了有效数据，0表示该行为空（未使用）。

- 系统启动时，所有Valid位初始化为0。

- 当某块首次被加载到Cache时，Valid位置1。

  

### 4.5 Cache访问流程（完整示例）

**PDF P20~25 示例**：

- Cache参数：8块，每块1字（4字节），直接映射，初始全部无效。

- 访问序列（字地址）：22, 26, 22, 16, 3, 16, 18。

  

**逐次跟踪**：

| 访问序号 | 地址 | 二进制（5位） | 索引（低3位） | Tag（高2位） | 结果 | Cache变化 |

| :--- | :--- | :--- | :--- | :--- | :--- | :--- |

| 1 | 22 | 10110 | 110 | 10 | 缺失 | 块6存入Tag=10 |

| 2 | 26 | 11010 | 010 | 11 | 缺失 | 块2存入Tag=11 |

| 3 | 22 | 10110 | 110 | 10 | **命中** | 无变化 |

| 4 | 16 | 10000 | 000 | 10 | **命中** | 无变化 |

| 5 | 3 | 00011 | 011 | 00 | 缺失 | 块3存入Tag=00 |

| 6 | 16 | 10000 | 000 | 10 | **命中** | 无变化 |

| 7 | 18 | 10010 | 010 | 10 | 缺失 | 块2被替换为Tag=10（旧Tag=11被丢弃） |

  

**关键观察**：

- 地址22和26的索引分别为6和2，无冲突。

- 地址16和18的索引分别为0和2，地址18的访问导致块2中的Tag=11（地址26）被替换。如果之后再次访问26，又会发生缺失——这就是**冲突缺失（Conflict Miss）**。

  
  

## 五、Cache缺失类型（3C模型）

  

| 缺失类型 | 英文 | 定义 | 能否避免 |

| :--- | :--- | :--- | :--- |

| **强制缺失** | Compulsory / Cold Miss | 首次访问某个块，该块从未在Cache中出现过 | 无法避免（除非无限大Cache） |

| **容量缺失** | Capacity Miss | Cache总容量不够，无法容纳程序的工作集，导致之前被替换的块再次被访问 | 增加Cache容量 |

| **冲突缺失** | Conflict / Collision Miss | 直接映射或组相联中，多个块竞争同一Cache位置（全相联Cache无此类缺失） | 增加关联度 |

  

**典型比例**（SPEC2000基准）：

- 强制缺失：~1%

- 容量缺失：~5-10%

- 冲突缺失：~20-30%

  
  

## 六、写策略（Write Policy）—— 高频考点

  

### 6.1 写命中（Write Hit）

  

| 策略 | 操作 | 优点 | 缺点 |

| :--- | :--- | :--- | :--- |

| **写直达（Write-Through）** | 同时更新Cache和主存 | 实现简单，主存总是最新 | 写带宽大，性能损失大 |

| **写回（Write-Back）** | 只更新Cache，标记该块为**脏（Dirty）**，替换时才写回主存 | 减少写带宽，性能好 | 硬件复杂，需维护Dirty位 |

  

### 6.2 写缺失（Write Miss）

  

| 策略 | 操作 | 适用场景 |

| :--- | :--- | :--- |

| **写分配（Write Allocate）** | 将缺失块加载到Cache，然后更新Cache | 写回Cache（利用空间局部性） |

| **写不分配（No-Write Allocate / Write Around）** | 直接写入主存，不加载到Cache | 写直达Cache（避免污染Cache） |

  

**标准组合**：

- **写直达 + 写不分配**（早期简单系统）

- **写回 + 写分配**（现代高性能系统）

  

### 6.3 写缓冲（Write Buffer）

- **用途**：缓解写直达的性能损失。

- **原理**：将待写入主存的数据先放入写缓冲，CPU无需等待主存写入完成即可继续执行。仅当写缓冲满时CPU才停顿。

  
  

## 七、Cache性能度量与计算（6.4）—— 必考计算

  

### 7.1 平均内存访问时间（AMAT，Average Memory Access Time）

$$\text{AMAT} = \text{Hit Time} + \text{Miss Rate} \times \text{Miss Penalty}$$

  

### 7.2 CPU时间公式（含Cache缺失）

$$\text{CPU时间} = (\text{CPI}_{\text{base}} + \text{每条指令平均缺失周期数}) \times \text{时钟周期}$$

其中：

$$\text{每条指令平均缺失周期数} = \text{指令缺失率} \times \text{缺失惩罚} + \text{数据访问比例} \times \text{数据缺失率} \times \text{缺失惩罚}$$

  

### 7.3 完整计算例题（PDF P37）

**给定**：

- 基础CPI（理想Cache）= 2

- I-Cache缺失率 = 2%

- D-Cache缺失率 = 4%

- 缺失惩罚 = 100个时钟周期

- Load/Store指令占全部指令的36%

  

**计算**：

- **指令缺失贡献**：$0.02 \times 100 = 2$ 周期/指令

- **数据缺失贡献**：$0.36 \times 0.04 \times 100 = 1.44$ 周期/指令

- **每条指令缺失周期** = $2 + 1.44 = 3.44$ 周期/指令

- **实际CPI** = $2 + 3.44 = \mathbf{5.44}$

- **性能比**（理想Cache vs 实际Cache）：$5.44 / 2 = \mathbf{2.72}$ 倍

  

**解读**：因为Cache缺失造成的停顿，使得该CPU的实际性能仅为理想情况下的 $1/2.72 \approx 36.8\%$。

  
  

## 八、关联度（Associativity）与替换策略

  

### 8.1 三种关联方式

| 类型 | 映射规则 | 硬件成本 | 缺失率 |

| :--- | :--- | :--- | :--- |

| **直接映射（1-路组相联）** | 每块只能放一个位置 | 最低（1个比较器） | 最高（冲突多） |

| **组相联（n-路）** | 每组n个位置可选 | 中等（n个比较器） | 中等 |

| **全相联** | 任何块可放任何位置 | 最高（所有块比较） | 最低（无冲突） |

  

- **组相联的计算**：

  $$\text{组号} = \text{块地址} \bmod \text{组数（\#Sets）}$$

  组内 $n$ 个候选位置可任意存放（由替换策略决定）。

  

### 8.2 关联度与缺失率的关系（实测数据，PDF P45）

64KB D-Cache，16字块，SPEC2000基准：

| 关联度 | 缺失率 |

| :--- | :--- |

| 1-路（直接映射） | 10.3% |

| 2-路组相联 | 8.6% |

| 4-路组相联 | 8.3% |

| 8-路组相联 | 8.1% |

  

**观察**：从1-路到2-路改善最明显（降低17%），再增加关联度收益递减。

  

### 8.3 替换策略

| 策略 | 原理 | 硬件实现 | 性能 |

| :--- | :--- | :--- | :--- |

| **LRU（最近最少使用）** | 替换最久未访问的块 | 2路易实现；4路尚可；更多路数成本高 | 较好 |

| **随机替换** | 随机选择一块替换 | 最简单（计数器+随机数） | 接近LRU（高关联度时） |

| **FIFO（先进先出）** | 替换最早加载的块 | 简单（队列） | 可能较差（循环访问时） |

  

**关联度对比例题（PDF P43~44）**：

访问序列（块地址）：0, 8, 0, 6, 8，4块Cache。

- **直接映射**（4块，块号=地址%4）：

  - 0→块0，8→块0（冲突，踢掉0），0→块0（缺失，重新加载），6→块2，8→块0（冲突，踢掉0）。缺失：0,8,0,6,8共5次。

- **2-路组相联**（2组，每组2块，组号=地址%2）：

  - 0→组0-路0，8→组0-路1，0命中，6→组2-路0，8命中。缺失：0,8,6共3次。

- **全相联**（1组，4块）：

  - 0,8,0命中,6,8命中。缺失：0,8,6共3次（容量缺失为主）。

- **结论**：增加关联度显著减少冲突缺失。

  
  

## 九、多级Cache（6.4）—— 高频综合计算

  

### 9.1 设计权衡

| 层级 | 设计目标 | 典型特征 |

| :--- | :--- | :--- |

| **L1 Cache** | 最小化**命中时间** | 小容量（16~64KB），小块大小（16~64B），写直达（简化设计） |

| **L2 / L3 Cache** | 最小化**缺失率**（避免访主存） | 大容量（数百KB~数十MB），大块大小（64~128B），写回 |

  

### 9.2 多级Cache性能计算

**完整例题（PDF P49~50）**：

- CPU频率：4GHz（$\text{周期} = 0.25\text{ns}$）

- 基础CPI = 1

- L1缺失率 = 2%（每指令）

- 主存访问时间 = 100ns = 400个周期

  

**场景一：只有L1 Cache**

- 缺失惩罚 = 400周期

- 实际CPI = $1 + 0.02 \times 400 = \mathbf{9}$

  

**场景二：增加L2 Cache**

- L2访问时间 = 5ns = 20个周期

- 全局主存缺失率 = 0.5%（即L1缺失后，再去L2，仍然缺失的比例）

- L2缺失惩罚（访主存）= 400周期

- **每条指令的缺失周期** = L1缺失造成的L2访问 + L2缺失造成的主存访问

  - 对L1缺失（2%的情况）：若在L2命中，惩罚20周期；若L2也缺失（0.5%全局），总惩罚 = 20 + 400 = 420周期。

  - 计算公式：$0.02 \times 20 + 0.005 \times 400 = 0.4 + 2 = 2.4$ 周期/指令

- 实际CPI = $1 + 2.4 = \mathbf{3.4}$

- **加速比** = $9 / 3.4 \approx \mathbf{2.6}$ 倍

  

**关键理解**：

- L2的**全局缺失率**（0.5%）是相对于全部指令访问而言的，不是L1缺失次数的25%（即0.02 × 0.25 = 0.005，正确）。

- 公式中 $0.005 \times 400$ 是主存访问贡献，$0.02 \times 20$ 是L2访问贡献。

  
  

## 十、虚拟内存（6.7）—— 硬件与OS协同

  

### 10.1 核心思想

- **主存作为磁盘的“Cache”**：将磁盘（存储设备）中的页（Page）缓存在主存中。

- **页面（Page）**：虚拟内存中的块，典型大小4KB。

- **缺页（Page Fault）**：请求的页不在主存中，需从磁盘加载，代价极其高昂（数百万个时钟周期）。

  

### 10.2 地址转换

- **虚拟地址空间**：每个进程独立拥有（32位→4GB，64位→16EB）。

- **物理地址空间**：实际的物理内存地址。

- **页表（Page Table）**：存储虚拟页号到物理页号（或磁盘位置）的映射表。

  

**地址划分**（以4KB页为例）：

$$\text{虚拟地址} = \text{虚拟页号（VPN）} + \text{页内偏移（Offset）}$$

VPN用于索引页表，Offset直接作为物理地址的低位。

  

### 10.3 页表项（PTE，Page Table Entry）

| 字段 | 作用 |

| :--- | :--- |

| **物理页号（PPN）** | 若页在主存中，存物理地址 |

| **有效位（Valid）** | 1表示页在主存中，0表示不在 |

| **引用位（Reference / Use）** | 1表示该页最近被访问过（用于LRU近似） |

| **脏位（Dirty）** | 1表示该页被写过（替换时需写回磁盘） |

| **保护位（Protection）** | 读/写/执行权限（用于内存保护） |

  

### 10.4 TLB（Translation Look-Aside Buffer）—— 关键加速部件

- **问题**：每次访存都要查页表（主存），额外增加一次内存访问。

- **解决方案**：页表项的小型Cache，称为TLB。

- **特性**：

  - 位于CPU内部

  - 命中时间：约1个时钟周期

  - 容量：16 ~ 512项

  - 缺失率：0.01% ~ 1%

  - 全相联或组相联

  

- **TLB缺失处理**：

  - **硬件管理**：CPU自动从页表加载PTE到TLB（复杂，x86采用）

  - **软件管理**：触发异常，由OS加载PTE到TLB（RISC-V采用，更灵活）

  

### 10.5 缺页处理（Page Fault Handler）

1. CPU访问虚拟地址，TLB缺失，查页表发现Valid=0。

2. 触发缺页异常，切换到操作系统内核模式。

3. OS从磁盘中找到该页（利用PTE中的磁盘地址）。

4. 在内存中选择一个物理页框替换（若被替换页Dirty=1，需先写回磁盘）。

5. 将磁盘页读入内存，更新页表（Valid=1）。

6. 返回到缺页的指令，重新执行（此时页已在内存中）。

  
  

## 十一、缓存一致性（6.10）—— 多核场景

  

### 11.1 问题定义

在多核处理器中，每个核心有自己的私有Cache，共享同一物理内存。当不同核心修改同一数据的不同副本时，会导致**不一致（Inconsistency）**。

  

### 11.2 一致性协议（Snooping Protocol）

- **思想**：所有Cache**监听（Snoop）** 总线上的读写操作，维护本地副本的有效性。

  

**写失效协议（Write Invalidate）** —— 最常用（如MESI协议）：

| 事件 | 行为 |

| :--- | :--- |

| 核心A写一块 | 广播**失效（Invalidate）** 信号 |

| 核心B的Cache监听到 | 若B有该块的副本，将其标记为**无效（Invalid）** |

| 核心B后续读该块 | 因块无效，发生Cache缺失，从核心A的Cache获取最新数据 |

  

**示例（PDF P96）**：

| 时间 | CPU活动 | 总线活动 | CPU A的Cache | CPU B的Cache | 主存 |

| :--- | :--- | :--- | :--- | :--- | :--- |

| 0 | 初始 | — | — | — | X=0 |

| 1 | CPU A读X | 缺失 | X=0（共享） | — | X=0 |

| 2 | CPU B读X | 缺失 | X=0（共享） | X=0（共享） | X=0 |

| 3 | CPU A写X=1 | 失效X | X=1（独占/脏） | X=0→无效 | X=0 |

| 4 | CPU B读X | 缺失（从A取） | X=1 | X=1 | X=1（A写回） |

- **关键**：在步骤3中，主存尚未更新，但CPU B通过监听协议从CPU A获取最新值。

  

### 11.3 一致性 vs 内存一致性

- **缓存一致性（Cache Coherence）** ：保证同一地址的读写顺序（单块数据）。

- **内存一致性（Memory Consistency）** ：决定不同地址的写操作何时对其他处理器可见（多个数据之间）。例如x86采用TSO（Total Store Order），而RISC-V采用弱一致性（需要`fence`指令显式同步）。

  
  

## 十二、软件优化：循环分块（Blocking）（6.4末）

  

### 12.1 问题来源：矩阵乘法（DGEMM）

```c

for (int i = 0; i < n; i++)

    for (int j = 0; j < n; j++)

        for (int k = 0; k < n; k++)

            C[i][j] += A[i][k] * B[k][j];

```

- B矩阵的列访问步幅为n（每列间隔n个元素），破坏空间局部性，导致大量Cache缺失。

- A矩阵的一行访问完所有列后，元素可能已被替换出Cache，后续对同一行的再次访问又是缺失。

  

### 12.2 解决方案：分块（Blocking）

- **思想**：将矩阵划分为小**子块（Block）**（如32×32），使子块能完全装入Cache。

- 在子块内部完成所有计算，数据被反复利用后才被替换。

  

**代码结构（PDF P56）**：

```c

#define BLOCKSIZE 32

void do_block(int n, int si, int sj, int sk, double *A, double *B, double *C) {

    for (int i = si; i < si + BLOCKSIZE; i++)

        for (int j = sj; j < sj + BLOCKSIZE; j++) {

            double cij = C[i + j*n];

            for (int k = sk; k < sk + BLOCKSIZE; k++)

                cij += A[i + k*n] * B[k + j*n];

            C[i + j*n] = cij;

        }

}

void dgemm(int n, double *A, double *B, double *C) {

    for (int sj = 0; sj < n; sj += BLOCKSIZE)

        for (int si = 0; si < n; si += BLOCKSIZE)

            for (int sk = 0; sk < n; sk += BLOCKSIZE)

                do_block(n, si, sj, sk, A, B, C);

}

```

- **效果**：Cache缺失率大幅降低，性能可提升数倍（根据矩阵规模，加速比可达3~10倍）。

  
  

## 十三、错误检测与纠正（SEC/DED码）（6.5）

  

### 13.1 汉明距离与纠错能力

- **汉明距离（Hamming Distance）** ：两个位模式之间不同的位数。

- **能力判定**：

  - 最小距离 $\ge$ 2：检测1位错误（如奇偶校验）

  - 最小距离 $\ge$ 3：纠正1位错误（SEC，Single Error Correction）

  - 最小距离 $\ge$ 4：纠正1位错误并检测2位错误（SEC/DED）

  

### 13.2 SEC/DED编码原理

- 在64位数据上增加**8位校验位**，总长度72位（ECC DRAM标准）。

- **数据位编号**：位置为2的幂（1, 2, 4, 8, ...）的位作为校验位。

- **解码**：校验位取值指示出错位的位置。若附加一位全局偶校验，可区分1位错误 and 2位错误。

  

### 13.3 应用

- **ECC DRAM**：用于服务器、工作站等高可靠性系统，可自动纠正单比特软错误。

  
  

## 十四、常见谬误与陷阱（必读！）

  

1. **陷阱：字节地址 vs 字地址的偏移计算**。

   - Cache的块索引计算基于**字节地址**。若块大小=16字节（4字），地址36（字节地址）→ 块号36//16=2；但若说“字地址36”，则块号=(36×4)//16=9。必须明确操作单位。

  

2. **陷阱：误认为“更大的块总是更好”**。

   - 大块利用空间局部性减少强制缺失，但也带来更大的缺失惩罚（传输更多数据）和可能的**污染（Pollution）** ——加载了大量程序中不使用或很少使用的数据，挤占了有用的Cache空间。对于步幅大、随机访问的程序，小块更优。

  

3. **陷阱：混淆L2的本地缺失率与全局缺失率**。

   - 全局缺失率 = L1缺失率 $\times$ L2本地缺失率（针对L1缺失的访问）。在多级Cache性能计算中，主存访问贡献必须使用**全局缺失率**。

  

4. **陷阱：写回Cache替换时忘记检查Dirty位**。

   - 若被替换的块是脏的（Dirty=1），必须先写回主存（或下一级Cache），然后才能丢弃该块，否则数据永久丢失。

  

5. **陷阱：将TLB缺失与缺页混为一谈**。

   - TLB缺失仅表示页表项不在TLB中，但页本身可能在主存中（Valid=1）。此时仅需从页表加载PTE到TLB（几十周期）。缺页（Page Fault）才是真正的磁盘访问（数百万周期）。

  

6. **陷阱：多核场景中认为“主存始终是最新数据”**。

   - 在写回Cache中，最新数据可能在某个核心的私有Cache中（脏块），尚未写回主存。这就是为什么需要缓存一致性协议（如MESI）来协调。

  

7. **难点：阻塞Cache vs 非阻塞Cache**。

   - **阻塞Cache**：缺失发生时，CPU停顿等待数据返回。计算AMAT足够。

   - **非阻塞Cache**：缺失期间允许后续命中指令继续执行（命中下命中，缺失下命中），性能分析需用模拟器，简单的AMAT公式不适用。

  
  

## 十五、总结性速查表

  

| 概念 | 关键公式 / 数值 | 核心要点 |

| :--- | :--- | :--- |

| **局部性** | 时间局部性 + 空间局部性 | 存储器层次结构的理论基础 |

| **磁盘访问时间** | 寻道 + 旋转延迟 + 传输 + 控制器 | 旋转延迟 = 0.5 × (1/转速) |

| **Cache索引** | 索引 = 块地址 mod #Blocks | 直接映射唯一位置 |

| **AMAT** | Hit Time + Miss Rate × Miss Penalty | 平均内存访问时间 |

| **CPU时间** | (CPI_base + 缺失周期/指令) × 时钟周期 | 含Cache缺失的实际性能 |

| **3C缺失** | 强制、容量、冲突 | 冲突缺失可通过增加关联度降低 |

| **写策略** | 写直达/写回；写分配/写不分配 | 现代系统：写回+写分配 |

| **TLB** | 页表项Cache，位于CPU内部 | 加速地址转换 |

| **缺页** | 磁盘访问，数百万周期 | 由OS处理，完全相联映射 |

| **一致性协议** | 监听（Snooping） | 写失效（Invalidate）最常用 |

| **分块优化** | 利用Cache重用，减少缺失 | 矩阵乘法性能提升数倍 |
